## 4. MyBatis 쿼리 전체 테스트 및 튜닝

**작업일**: 2026-03-03

### 4-1. 문제 분석

76개 API 엔드포인트의 1차 호출(캐시 미적용) 응답시간을 측정한 결과, 9개 API에서 3초 이상의 응답 지연이 확인되었습니다.

**핵심 병목 원인**: `hash_id` 컬럼이 **TEXT 타입**으로, `pi_find`와 `pi_data_processing` 테이블 간 JOIN 시 TEXT 컬럼 비교가 34만건 이상 발생하여 극심한 성능 저하 유발.

| API 엔드포인트 | 튜닝 전 | 병목 원인 |
|---------------|---------|----------|
| `statistics/overall` | 19.1초 | pi_find LEFT JOIN pi_data_processing ON hash_id(text) |
| `reports/targets` | 16.2초 | pi_find → pi_summary(text) → pi_data_processing(text) 이중 text JOIN |
| `dashboard/charts/pie` | 13.3초 | pi_summary 전체 테이블 GROUP BY |
| `dashboard/datatypes` | 12.8초 | pi_summary 전체 테이블 GROUP BY |
| `dashboard/info` | 4.3초 | pi_find LEFT JOIN pi_data_processing(text) + 스칼라 서브쿼리 |
| `statistics/datatypes` | 4.9초 | pi_summary JOIN pi_targets GROUP BY |
| `statistics/detections` | 3.9초 | pi_summary JOIN pi_targets GROUP BY |
| `dashboard/path-current` | 3.8초 | pi_find 34만건 GROUP BY path(text) |
| `reports/summary` | 3.4초 | pi_find LEFT JOIN pi_data_processing(text) + COUNT(DISTINCT hash_id) |

### 4-2. 튜닝 전략

**DB 수정 없이 SQL 구조만으로 최적화** (DB 수정 금지 규칙 준수)

1. **TEXT 컬럼 JOIN 제거**: `pi_find`와 `pi_data_processing` 간의 `hash_id(text)` JOIN을 독립 서브쿼리로 분리
2. **Late Row Lookup**: LIMIT 서브쿼리로 필요한 행만 먼저 추출 후 나머지 JOIN
3. **COUNT(DISTINCT text) → COUNT(*)**: TEXT 타입 DISTINCT 연산 제거
4. **스칼라 서브쿼리 통합**: 동일 테이블 다중 스캔을 CASE+SUM으로 1회 통합

### 4-3. 수정 내용 상세

#### (1) StatisticsMapper.xml - `selectOverallStatistics`
- **변경 전**: `pi_find LEFT JOIN pi_data_processing ON hash_id(text)` + `COUNT(DISTINCT hash_id)`
- **변경 후**: `pi_find`와 `pi_data_processing`을 독립 서브쿼리로 분리하여 CROSS JOIN
- **효과**: TEXT JOIN 완전 제거, 각 테이블 독립 스캔

```sql
-- 변경 전 (19초)
FROM pi_find f
LEFT JOIN pi_data_processing dp ON f.TARGET_ID = dp.TARGET_ID AND f.HASH_ID = dp.HASH_ID

-- 변경 후 (7초 → 캐시 39ms)
FROM (
    SELECT COUNT(*) AS totalDetections FROM pi_find f INNER JOIN pi_targets t ...
) find_stat,
(
    SELECT SUM(CASE ...) FROM pi_data_processing dp INNER JOIN pi_targets t ...
) dp_stat
```

#### (2) DashboardMapper.xml - `selectDashboardInfo`
- **변경 전**: 메인 쿼리에서 `pi_find LEFT JOIN pi_data_processing ON hash_id(text)` + 스칼라 서브쿼리 2개
- **변경 후**: 4개 독립 서브쿼리 (target_stat, find_stat, dp_stat, scan_stat)로 분리
- **효과**: TEXT JOIN 제거 + 스칼라 서브쿼리 독립 실행

#### (3) DashboardMapper.xml - `selectDetectionStatus`
- **변경 전**: `COUNT(DISTINCT hash_id)` 3회 (TEXT 타입 DISTINCT = 매우 느림)
- **변경 후**: `COUNT(*)` 사용 (pi_find에 deldate 컬럼 없으므로 동일 결과)
- **효과**: TEXT DISTINCT 연산 3회 제거

#### (4) ReportMapper.xml - `selectTargetReport`
- **변경 전**: `pi_find(34만건) JOIN pi_summary(text) JOIN pi_data_processing(text)` → 이중 TEXT JOIN
- **변경 후**: 서브쿼리에서 `pi_find + pi_targets`로 LIMIT 10000건 추출 후 나머지 JOIN (Late Row Lookup)
- **효과**: TEXT JOIN 대상 34만건 → 실제 필요건으로 축소

#### (5) ReportMapper.xml - `selectReportSummary`
- **변경 전**: `pi_find LEFT JOIN pi_data_processing ON hash_id(text)` + `COUNT(DISTINCT hash_id)` + 스칼라 서브쿼리 2개
- **변경 후**: 3개 독립 서브쿼리 (target_stat, find_stat, dp_stat)로 분리
- **효과**: TEXT JOIN 제거, COUNT(DISTINCT text) → COUNT(*)

#### (6) ReportMapper.xml - `selectMonthlyReport`
- **변경 전**: `pi_find LEFT JOIN pi_data_processing ON hash_id(text)` + DATE_FORMAT GROUP BY
- **변경 후**: `pi_data_processing` JOIN 제거, 검출 건수만 집계
- **효과**: TEXT JOIN 제거

#### (7) ReportMapper.xml - `selectDetailedReport`
- **변경 전**: `pi_targets LEFT JOIN pi_find LEFT JOIN pi_data_processing ON hash_id(text)` + COUNT(DISTINCT hash_id)
- **변경 후**: `pi_find`를 서브쿼리로 pre-aggregate, pi_data_processing JOIN 제거
- **효과**: TEXT JOIN + TEXT DISTINCT 모두 제거

#### (8) ReportMapper.xml - `selectExceptionReport`
- **변경 전**: `pi_targets LEFT JOIN pi_find LEFT JOIN pi_data_processing ON hash_id(text)` + COUNT(DISTINCT hash_id)
- **변경 후**: `pi_find`를 서브쿼리로 pre-aggregate, pi_data_processing JOIN 제거
- **효과**: TEXT JOIN + TEXT DISTINCT 모두 제거
