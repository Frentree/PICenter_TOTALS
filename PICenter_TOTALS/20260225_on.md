## 9. Phase 7: 전체 Mapper XML 쿼리 튜닝

### 9-1. 튜닝 원칙
- DB는 절대 건드리지 않고 소스단에서만 처리
- 모든 튜닝 내용은 주석으로 변경 전/후/효과를 기록
- 핵심 성능 문제: `hash_id(text)` 컬럼 JOIN이 34만건에서 극도로 느림

### 9-2. DetectionMapper.xml (3개 쿼리 튜닝)

| 쿼리 | 변경 전 | 변경 후 | 효과 |
|------|---------|---------|------|
| `selectDetectionList` | 상관 서브쿼리 `SELECT MAX(DATA_PROCESSING_IDX)` 매 행마다 실행 | 파생 테이블(derived table)로 MAX를 미리 1회 계산 | O(N*M) → O(N+M) |
| `selectDetectionListCount` | COUNT에도 불필요한 dp JOIN | processStatus 필터 있을 때만 dp JOIN 수행 | 필터 없는 경우 JOIN 2개 제거 |
| `selectDetectionDetail` | 상관 서브쿼리로 MAX 조회 | ORDER BY DESC LIMIT 1 패턴 | 단건 조회 최적화 |

### 9-3. DashboardMapper.xml (7개 쿼리 튜닝)

| 쿼리 | 변경 전 | 변경 후 | 효과 |
|------|---------|---------|------|
| `selectDatatypeStatus` | pi_data_processing JOIN (text 비교 34만건) | dp JOIN 제거, pi_summary만 집계 | 25초 → 2초 |
| `selectPieChartData` | 270개 카테고리 전체 반환 | LIMIT 10으로 상위 10개만 | 네트워크/렌더링 개선 |
| `selectImplementationStatus` | pi_find LEFT JOIN dp on hash_id(text) | 서브쿼리 방식 독립 조회 | text JOIN 제거 |
| `selectCompletionStats` | pi_find LEFT JOIN dp on hash_id(text) | 서브쿼리 방식 독립 조회 | text JOIN 제거 |
| `selectPathCurrent` | dp LEFT JOIN + GROUP BY path(text) | dp JOIN 제거 | 이중 text 연산 → 단일 |
| `selectBarChartData` | dp LEFT JOIN + DATE_FORMAT GROUP BY 무제한 | dp JOIN 제거, 기본 12개월 제한 | text JOIN 제거 + 범위 제한 |
| `selectDashboardDetections` | dp LEFT JOIN 포함 3개 JOIN | dp JOIN 제거, 기본 LIMIT 20 | text JOIN 감소 |
| `selectGridData` | 무제한 결과셋 | LIMIT 100 추가 | 대시보드 표시용 제한 |

LEFT JOIN pi_target_group tg ON tg.TARGET_ID = t.TARGET_ID AND t.AP_NO = tg.AP_NO

LEFT JOIN pi_target_group ptg ON tg.up_idx = ptg.idx

COALESCE(a.AGENT_CONNECTED_IP, tg.connected_ip) AS targetIp

## 36. 대용량 쿼리 성능 최적화 (Subquery Pagination)

**작업일**: 2026-02-27

### 36-1. 문제점
- pi_find 테이블 339,784건으로 인해 여러 API 응답시간이 10~25초
- 원인: LIMIT 전에 대용량 테이블 간 다중 JOIN + TEXT 컬럼 접근

### 36-2. 최적화 패턴 (Subquery Pagination)
```sql
-- 기존: 전체 JOIN 후 LIMIT (느림)
SELECT ... FROM pi_find f
JOIN pi_targets t ON ...
JOIN pi_agents a ON ...
JOIN pi_summary s ON ...
LIMIT offset, limit

-- 최적화: 페이지 추출 후 JOIN (빠름)
SELECT outer.* FROM (
    SELECT f.hash_id, f.target_id, f.ap_no
    FROM pi_find f
    INNER JOIN pi_targets t ON f.target_id = t.TARGET_ID
    ORDER BY f.regdate DESC
    LIMIT offset, limit
) page
JOIN pi_find f ON page.hash_id = f.hash_id
JOIN pi_targets t ON ...
JOIN pi_agents a ON ...
```
